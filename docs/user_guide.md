# Руководство пользователя

## Первый запуск

1. Создайте файл `DocumentControl.xlsm`.
2. Импортируйте VBA из папок `vba/modules`, `vba/classes`, `vba/forms` (см. `docs/forms_import.md`).
3. Запустите `modMain.AppInitialize` — будут созданы все листы, заголовки и базовые справочники.
4. Убедитесь, что в листе `cfg_app` заданы пути:
   - `templates_path` — папка с файлами `RepairInstruction.dotx` / `EngineeringAnalysis.dotx`
   - `output_path` — папка, куда будут сохраняться DOCX и PDF

---

## Сценарий 1: Создание документа RI

1. Нажмите кнопку **Новая карточка** на `ui_dashboard` (или запустите `modMain.OpenDocumentCard`).
2. В открывшейся форме заполните обязательные поля (выделены ниже).
3. Нажмите **Create DOCX** — система создаёт документ из `RepairInstruction.dotx` и подставляет все реквизиты.
4. Откройте созданный DOCX и заполните вручную инженерное содержание в разделах:
   - `3.1 Описание повреждения`
   - `3.2 Описание ремонта`
   - `5.1 Выполнение осмотра`
   - `5.2 Заключительные работы`
   - Таблицу `6. Инструкции по поддержанию летной годности`
5. Не удаляйте заголовки разделов и не трогайте таблицу применяемости — реквизиты уже подставлены системой.

## Сценарий 2: Предвыпускная проверка

1. В форме карточки нажмите **Validate**.
2. Откроется `frmValidationReport` с перечнем ошибок (ERROR) и предупреждений (WARNING) и счётчиками в заголовке.
3. Исправьте ошибки в Word и/или в карточке.
4. Запустите проверку повторно.

## Сценарий 3: Выпуск PDF

1. После прохождения проверки без ошибок нажмите **Export PDF**.
2. Путь к PDF записывается в карточку автоматически.

## Поля формы `frmDocumentCard`

### Идентификация документа

| Поле | Описание | Обязательно |
|---|---|---|
| `Document ID` | Уникальный номер документа, напр. `RI-2026-001` | ✅ |
| `Document Type` | `Repair Instruction` или `Engineering Analysis` | ✅ |
| `Title` | Краткий технический заголовок ремонта | ✅ |
| `Revision` | Ревизия документа; при первом издании — `–` (тире) | ✅ |
| `Date` | Дата документа в формате `DD.MM.YYYY`, напр. `25.03.2026` | ✅ |
| `Status` | `Draft` / `In Review` / `Released` | — |

### Воздушное судно

| Поле | Описание | Обязательно |
|---|---|---|
| `Aircraft Model (Type)` | Тип ВС, напр. `RRJ-95`, `A320` | ✅ |
| `Aircraft Variant/Model` | Модель/вариант, напр. `RRJ-95B100` | — |
| `Aircraft Reg. Number` | Регистрационный номер, напр. `RA-89001` | ⚠ |
| `MSN` | Заводской серийный номер | ⚠ |
| `Aircraft Manuf. Date` | Дата изготовления ВС | — |
| `Aircraft Hours (FH)` | Наработка ВС в лётных часах | — |
| `Aircraft Cycles (FC)` | Наработка ВС в лётных циклах | — |

### Компонент/узел

| Поле | Описание | Обязательно |
|---|---|---|
| `Assembly Number` | Обозначение компонента (топ-уровень), напр. `SML4000-60-0` | ⚠ |
| `Part Number` | Обозначение повреждённой детали, напр. `SML4042-02-0` | — |
| `Component Name` | Наименование компонента, напр. `PIVOTING DOOR RH` | ✅ |
| `Component S/N` | Серийный номер компонента | — |
| `Component Hours (FH)` | Наработка компонента в лётных часах | — |
| `Component Cycles (FC)` | Наработка компонента в лётных циклах | — |
| `Component Manuf. Date` | Дата изготовления компонента | — |
| `Applicability` | Применяемость (условия/ограничения) | — |

### Ответственные лица

| Поле | Описание | Обязательно |
|---|---|---|
| `Author` | Автор (инженер-разработчик) | ✅ |
| `Checker` | Проверяющий | ⚠ |
| `Approver` | Утверждающий | ⚠ |

### Перекрёстные ссылки и прочее

| Поле | Описание |
|---|---|
| `Related Analysis #` | Номер связанного Engineering Analysis |
| `Related Instruction #` | Номер связанного Repair Instruction |
| `References` | Ссылочные документы (SDR, CMM, AMM и т.п.) |
| `Attachments` | Перечень вложений |
| `Remarks` | Примечания |

### Служебные (заполняются автоматически)

| Поле | Описание |
|---|---|
| `Word Doc Path` | Путь к сгенерированному DOCX |
| `PDF Path` | Путь к сгенерированному PDF |

> ✅ — блокирующая ошибка при валидации  
> ⚠ — предупреждение при валидации

## Кнопки формы

| Кнопка | Действие |
|---|---|
| **Save Card** | Сохраняет/обновляет карточку на листе `doc_cards` |
| **Create DOCX** | Создаёт Word-документ из шаблона, подставляет все маркеры, записывает `Word Doc Path` |
| **Validate** | Запускает 14 проверок и показывает `frmValidationReport` |
| **Export PDF** | Экспортирует DOCX в PDF, записывает `PDF Path` |
| **Close** | Закрывает форму |
| **Field Help** | Показывает встроенную подсказку по всем полям и кнопкам |

## Что создаёт шаблон RI

При нажатии **Create DOCX** система открывает `RepairInstruction.dotx` (на базе `A.095.04.00000.078.0699_example.docx`) и подставляет реквизиты во все маркеры:

- Титульная страница: тип ВС + заголовок, номер документа, ревизия
- Таблица подписей: автор, проверяющий, утверждающий
- Лист регистрации изменений: ревизия, дата
- Таблица применяемости (раздел 4.2): все данные ВС и компонента
- Раздел 7 (Уведомление о выполнении ремонта): номер и заголовок документа
- Колонтитул каждой страницы: номер документа, ревизия, дата

Разделы `3`, `5` и остальные инженерные секции содержат текст `<<Заполняется инженером>>` — их инженер заполняет вручную.

## Типовые ошибки и решения

| Ошибка | Причина | Решение |
|---|---|---|
| `Template file not found` | Неверный `templates_path` в `cfg_app` или файл отсутствует | Проверить путь и наличие `.dotx` файла |
| `Template is not a valid OOXML file` | Шаблон — текстовый файл, а не ZIP/OOXML | Заменить шаблон на корректный `.dotx` |
| `word_doc_path is empty` при экспорте | DOCX ещё не создан | Сначала нажать **Create DOCX** |
| Маркеры `{{...}}` остались в документе | Шаблон содержит маркер, которого нет в `ReplaceAllMarkers` | Уточнить список маркеров у разработчика |
| `<<Заполняется инженером>>` остался | Инженер не заполнил секцию | Заполнить вручную в Word |
