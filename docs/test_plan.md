# План тестирования

## Предусловия

- `DocumentControl.xlsm` создан и `AppInitialize` выполнен.
- В `cfg_app` заданы `templates_path` и `output_path`.
- `templates/RepairInstruction.dotx` присутствует и является валидным OOXML (сигнатура `PK`).
- Microsoft Word 2016+ установлен и COM-доступен.

---

## T1. Позитивные сценарии — Repair Instruction

### T1-1. Создание DOCX из шаблона

**Шаги:**
1. Открыть `frmDocumentCard`.
2. Заполнить все обязательные поля (см. таблицу полей ниже).
3. Нажать **Create DOCX**.

**Ожидаемый результат:**
- Файл `<DocumentID>_Rev<Revision>.docx` создан в `output_path`.
- Поле `Word Doc Path` в форме заполнено.
- В `log_actions` появилась запись `CreateWordDocument | OK`.

### T1-3. Экспорт PDF

**Шаги:**
1. После T1-1, нажать **Export PDF**.

**Ожидаемый результат:**
- Файл `<DocumentID>_Rev<Revision>.pdf` создан.
- Поле `PDF Path` в форме заполнено.
- В `log_actions` запись `ExportCurrentToPdf | OK`.

### T1-4. Успешная валидация

**Шаги:**
1. После T1-1, заполнить все инженерные секции в DOCX вручную (убрать `<<Заполняется инженером>>`).
2. Нажать **Validate**.

**Ожидаемый результат:**
- `frmValidationReport` открывается.
- Сообщение: `No issues found — document passed validation`.
- В `validation_issues` нет записей с `severity = ERROR`.

## T2. Негативные сценарии

### T2-1. Пустые обязательные поля

**Шаги:** Открыть форму, не заполнять поля, нажать **Validate**.

**Ожидаемый результат:** Ошибки ERROR для: `document_id`, `document_type`, `title`, `revision`, `date`, `aircraft_model`, `component_name`, `author`.

### T2-2. Шаблон не найден

**Шаги:** Задать в `cfg_app` несуществующий `templates_path`, нажать **Create DOCX**.

**Ожидаемый результат:** `MsgBox` с текстом `Template file not found:`.

### T2-3. Шаблон не является OOXML

**Шаги:** Подменить `.dotx` на текстовый файл, нажать **Create DOCX**.

**Ожидаемый результат:** `MsgBox` с текстом `Template is not a valid OOXML file`.

### T2-4. Незаменённые маркеры в документе

**Шаги:** Вручную вставить `{{TestMarker}}` в DOCX, запустить **Validate**.

**Ожидаемый результат:** Ошибка ERROR `UNRESOLVED_MARKER`.

### T2-5. Незаполненные инженерные секции

**Шаги:** Не трогать `<<Заполняется инженером>>`, запустить **Validate**.

**Ожидаемый результат:** Предупреждение WARNING `TRASH_PLACEHOLDER`.

### T2-6. Слово "draft" в пути к файлу (регрессия)

**Шаги:** Задать `output_path = C:\projects\draft_docs`, создать DOCX, запустить **Validate**.

**Ожидаемый результат:** Предупреждение `TRASH_PLACEHOLDER` **не** появляется из-за слова "draft" в пути.

### T2-7. Export PDF без DOCX

**Шаги:** Очистить `Word Doc Path`, нажать **Export PDF**.

**Ожидаемый результат:** `MsgBox` с текстом `Word Doc Path is empty`.

## T3. Регрессия

| Тест | Критерий |
|---|---|
| Повторная генерация DOCX | Исходный `.dotx` шаблон не изменён (дата изменения не обновилась) |
| Журналирование | Все операции Save/Create/Validate/Export отражены в `log_actions` |
| Сохранение карточки | После Save данные на листе `doc_cards` соответствуют введённым (все 31 поле) |
| Загрузка строки | При открытии формы с курсором на строке 3+ данные загружаются в правильные поля |

---

## Чек-лист ручной приёмки

- [ ] `AppInitialize` создаёт все 14 листов с правильными заголовками (31 колонка в `doc_cards`).
- [ ] Карточка RI создаётся, сохраняется, загружается из `doc_cards`.
- [ ] DOCX создаётся из `RepairInstruction.dotx` (не из текстового файла).
- [ ] Все 20 маркеров заменены корректными значениями.
- [ ] Колонтитул содержит номер документа, ревизию, дату.
- [ ] Таблица применяемости (раздел 4.2) заполнена данными компонента.
- [ ] Валидация показывает понятные ошибки/предупреждения с кодами.
- [ ] `frmValidationReport` закрывается кнопкой **Close**.
- [ ] PDF успешно экспортируется.
- [ ] `log_actions` содержит записи всех ключевых действий.
- [ ] Процесс `WINWORD.EXE` завершается после каждой операции.
