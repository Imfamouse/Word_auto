# AGENTS.md

## Mission
Построить локальную систему подготовки инженерных документов для авиационной компании на базе Excel + Word + VBA.

Система НЕ должна автоматически сочинять инженерное содержание документа.
Система должна автоматизировать:
- единые реквизиты и повторяющиеся данные,
- выбор и применение шаблонов,
- проверку структуры документа,
- проверку согласованности данных,
- поиск типовых механических ошибок,
- выпуск итогового DOCX/PDF,
- журналирование действий пользователя.

## Core principle
Word остаётся средой, где инженер вручную пишет техническое содержание.
Excel является источником структурированных данных, правил и матриц проверки.
VBA является связующим слоем между Excel и Word.

## Domain constraints
В проекте есть как минимум два типа документов:
- Repair Instruction
- Engineering Analysis

Документы однотипны по процессу выпуска, но сильно различаются по инженерному содержанию.
Поэтому запрещено пытаться хранить весь текст документа в Excel или генерировать содержательную инженерную часть автоматически.

## Technical constraints
- Целевая среда: Windows + Excel + Word + VBA.
- Основа реализации: Excel VBA + Word Automation via COM.
- Решение должно работать локально, без обязательной серверной части.
- Код должен быть пригоден для поддержки несколькими разработчиками.
- Критичные правила не смешивать с UI-кодом.
- Пути к шаблонам и папкам хранения не хардкодить: использовать конфигурацию.
- Аккуратно освобождать COM-объекты Word и не оставлять висящие процессы.
- Не повреждать исходные шаблоны Word.

## Architecture constraints
Разделять:
- UI,
- доступ к данным Excel,
- бизнес-правила,
- автоматизацию Word,
- валидацию,
- отчётность,
- журналирование.

Не делать один огромный модуль.
Использовать понятные имена процедур, функций, форм и классов.
Использовать константы для имён листов, маркеров, статусов и кодов ошибок.

## Deliverables
В репозитории должны появиться:
- структура Excel-книги,
- VBA-модули, классы и формы,
- Word-шаблоны,
- документация для пользователя,
- документация для разработчика,
- тестовый план,
- пример тестовых данных,
- список ограничений и направлений развития.

## Working style
Работать по этапам:
1. Анализ и архитектура
2. MVP
3. Усиление проверок
4. UX и сопровождаемость
5. Тесты и документация

После каждого этапа:
- кратко объяснять, что сделано,
- перечислять изменённые файлы,
- фиксировать открытые риски,
- явно помечать предположения как ASSUMPTIONS.

## Hard rule
Do not try to generate engineering content.
This is not a document-writing AI.
This is a document-control and validation system.
The engineer writes the technical content manually in Word.
The software manages metadata, templates, consistency, and release checks.
