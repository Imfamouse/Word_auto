# Проект автоматизации инженерных документов

## Назначение
Проект предназначен для снижения количества механических ошибок при выпуске инженерных документов в авиационной компании.

Система должна:
- создавать документ из правильного Word-шаблона,
- подставлять единые реквизиты из Excel,
- помогать инженеру работать в привычном Word,
- проверять комплектность и согласованность документа перед выпуском,
- формировать DOCX/PDF и журнал действий.

## Почему это нужно
Сейчас документы часто создаются на основе старых файлов и вручную правятся по месту.
Из-за этого возникают типовые ошибки:
- в разных местах документа остаются разные номера,
- забывают поменять старый part number / assembly number / aircraft number,
- пропускают обязательные разделы,
- остаются заглушки типа TBD / XXX / ???,
- по Engineering Analysis часть пунктов базиса остаётся без явного закрытия.

## Ключевая идея
Система НЕ заменяет инженерное мышление.
Она убирает механические ошибки и контролирует структуру выпуска.

Excel:
- карточка документа,
- справочники,
- матрицы проверки,
- правила,
- журнал.

Word:
- шаблон,
- титул, шапка, колонтитулы,
- обязательные разделы,
- свободные зоны для инженерного текста.

VBA:
- связывает Excel и Word,
- создаёт документ,
- подставляет реквизиты,
- запускает проверки,
- формирует отчёт,
- сохраняет DOCX/PDF.

## Основные типы документов
- Repair Instruction
- Engineering Analysis

## Структура root-файлов
- AGENTS.md — обязательные инструкции для Codex
- PROJECT_BRIEF.md — краткий бизнес-контекст и рамки
- DOCUMENT_TYPES.md — описание типов документов
- VALIDATION_RULES.md — правила проверки
- IMPLEMENTATION_PLAN.md — этапы реализации
- NAMING_RULES.md — правила именования и хранения файлов
- CODEX_TASK.md — стартовый запрос для Codex

## Принцип работы с Codex
1. Codex читает AGENTS.md.
2. Затем использует root-документы как локальную документацию проекта.
3. Сначала проектирует архитектуру.
4. Потом делает MVP.
5. Потом усиливает проверки.
6. Потом улучшает UX и документацию.
